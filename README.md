# Análise comparativa pessoal

## Questão 1

Nessa questão, eu achei que a linguagem mais fácil foi Python, porque o código fica menor e mais simples de entender.

No Java e Swift também ficaram fáceis de ler, porque usam o `while` de uma forma parecida com o pseudocódigo. A diferença é que elas usam chaves e uma estrutura mais organizada.

A linguagem que eu achei mais difícil foi Haskell, porque ela usa uma forma diferente de pensar, com recursão. Para mim, isso não é tão natural quanto usar um `while`.

Na minha opinião, Python foi a melhor opção nessa questão, porque juntou facilidade de escrita com boa leitura.

## Questão 2

Nessa questão, eu achei Ruby mais fácil de entender. O `case when` ficou bem limpo e não precisou usar `break`.

Em C, o `switch` também ficou fácil de acompanhar, mas achei mais trabalhoso porque precisa colocar `break` nos casos. Se esquecer o `break`, o código pode funcionar errado.

Em Erlang, eu achei a escrita diferente, porque a linguagem trabalha de uma forma mais funcional. Mesmo assim, deu para entender a lógica depois de observar melhor.

Na minha opinião, Ruby ficou melhor nessa questão, porque o código ficou mais curto e mais claro.

## Questão 3

Nessa questão, achei mais complicado tirar o `break` sem mudar o funcionamento do código.

A solução foi usar uma variável de controle para dizer quando o laço deveria continuar ou parar. Isso deixou o código um pouco maior, mas evitou o uso do `break`.

Python foi a linguagem que eu achei mais fácil de escrever, porque tem menos símbolos e a leitura fica mais direta.

No JS ficou parecido com C, então também foi fácil de comparar com o código original. Já Kotlin ficou bem organizado por causa do `when`.

Na minha opinião, Python foi mais simples, mas Kotlin ficou mais organizado visualmente.

## Questão 4

Nessa questão, percebi que C e JS são as linguagens mais parecidas com Java, porque permitem colocar a inicialização, a condição e o incremento no próprio `for`.

Python, Ruby e Swift ficaram mais simples visualmente, mas nelas o valor de `j` precisou ser diminuído dentro do laço.

Achei Python o mais fácil de ler, porque o código fica curto e direto. Porém, se for comparar com o código original em Java, JavaScript ficou mais parecido.

No geral, achei essa questão mais tranquila, porque a lógica do `for` era simples e dava para adaptar bem para outras linguagens.
